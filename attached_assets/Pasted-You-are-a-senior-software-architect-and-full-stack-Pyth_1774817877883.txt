You are a senior software architect and full-stack Python developer.
Design and implement a Manufacturing Planning + Inventory + Location Transfer + User Management System. The base requirements are derived from the following specification

GOAL:

Build a complete system that supports these core workflows:

Manufacturing Plan (BOM-based):
I enter a Product Code and a Quantity to manufacture, and the system outputs the required raw materials with each material’s Code, Name, and Required Quantity.
Warehouse Inventory:
I can add/import materials into a main warehouse by entering Material Code, Material Name, Quantity, and optional Unit.
Stock Transfer Between Locations:
I can transfer a material from one location to another. I type/select the material by code or name, the system auto-fills its details, I enter transfer quantity and choose destination location. The quantity is deducted from the source location and added to the destination location.
Inventory Counting / Stock Audit Page:
Create a dedicated Inventory Count / Stock Audit page where users can:
view current stock by location
perform physical stock counting
compare system quantity vs actual counted quantity
see variance/difference
save stock adjustments
automatically generate an adjustment transaction in the audit log with tx_type = "ADJUST"
add notes for each adjustment
This page should be practical, clear, and suitable for real warehouse operations.
Admin / Manager Page:
Create a separate Admin Dashboard page for the system manager/administrator where they can:
add new users
edit users
activate/deactivate users
assign roles and permissions
control which pages/features each user can access
SYSTEM REQUIREMENTS
A) Data Model (Database)

Use SQLite for now, with architecture ready for PostgreSQL later if needed.

Create these tables:

1) products
id (PK)
product_code (unique)
product_name (optional)
2) materials
id (PK)
material_code (unique)
material_name
unit (optional, e.g. pcs, kg, meter)
3) bom_items
id (PK)
product_id (FK -> products.id)
material_id (FK -> materials.id)
qty_per_unit
4) locations
id (PK)
location_name (unique)
5) inventory_balances
id (PK)
material_id (FK -> materials.id)
location_id (FK -> locations.id)
quantity

6) inventory_transactions
id (PK)
tx_type (IN, OUT, TRANSFER, ADJUST)
material_id
from_location_id (nullable)
to_location_id (nullable)
quantity
tx_datetime
reference (optional)
user_note (optional)
7) users
id (PK)
username (unique)
full_name
password_hash
role
is_active
created_at

B) Workflow 1: Manufacturing Plan (BOM Explosion)

User inputs:

product_code
target_quantity_to_produce

System outputs:

material_code
material_name
required_qty = qty_per_unit × target_quantity_to_produce

Also show:

total number of materials
optional availability check from selected location
available_qty
shortage_qty = max(0, required_qty - available_qty)

Validation:

product_code must exist
target_quantity_to_produce > 0
BOM must exist for selected product
support decimal quantities where needed
C) Workflow 2: Warehouse Inventory Input (Stock In)

User inputs:

material_code
material_name
quantity
location (default: Main Warehouse)

Behavior:

If material_code exists, auto-fill material_name
Prevent conflicting name changes unless confirmed
If material does not exist, create it
Add quantity to inventory_balances
Create transaction record with tx_type = "IN"

Validation:

quantity > 0
location must exist or be created
no invalid negative quantities
D) Workflow 3: Transfer Materials Between Locations

User inputs:

material_code or material_name search
transfer_quantity
from_location
to_location

Behavior:

autocomplete for material code/name
auto-fill selected material details
check available stock in source location
if enough stock:
subtract from source
add to destination
write TRANSFER transaction
if not enough:
show validation error

Validation:

transfer_quantity > 0
from_location != to_location
material must exist
cannot transfer more than available
E) Workflow 4: Inventory Count / Stock Audit

User inputs:

location
material search/filter
actual counted quantity
note/reason

Behavior:

display system quantity
user enters actual counted quantity
calculate variance:
variance = actual_counted_qty - system_qty
if variance exists:
allow saving adjustment
update inventory balance
create ADJUST transaction in audit log
support filtering by location and material
support counting all materials in one location in a grid/table form

Validation:

counted quantity cannot be empty
support decimal values
save user and timestamp for each adjustment
F) Workflow 5: Admin / User Management

Create a separate page for administrators only.

Features:

create new user
edit user profile
reset/change password
activate/deactivate user
assign role
assign detailed permissions
restrict page access depending on role/permissions
hide unauthorized pages/buttons from users without permission

Suggested roles:

Admin: full access
Manager: can manage stock, reports, and manufacturing plans
Storekeeper: can stock in, transfer, and inventory count
Viewer: read-only access

Security:

store passwords as hashed values, never plain text
implement login page
session or state-based authentication for desktop app
permission checks at UI level and service level
